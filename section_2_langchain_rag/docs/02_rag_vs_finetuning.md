# Retrieval-Augmented Generation and How It Compares to Fine-Tuning

## The problem RAG was invented to solve

A large language model is only as knowledgeable as the text it was trained on, and once training is finished the model's world view is frozen. If a new product launches next week, the model will not know about it. If a company's internal policy document changes tomorrow, the model will still repeat the old version. Worse, when asked about something outside its knowledge, an LLM does not usually admit ignorance — it invents a plausible answer, a behaviour known as hallucination. Retrieval-Augmented Generation, almost always shortened to RAG, is the pattern that fixes both of these problems by giving the model a way to look things up at answer time instead of relying only on what is baked into its weights.

The idea is straightforward. Before the model generates its answer, a separate retrieval system searches a collection of documents for passages that seem relevant to the user's question. Those passages are then inserted into the prompt, together with an instruction that tells the model to use them as its source of truth. The model still generates the final text, but it is now grounded in evidence that was fresh at the moment of the query. If the evidence changes, so does the answer, and no retraining is needed.

## The RAG pipeline in plain terms

Every RAG system has the same shape. First, the source documents are chunked into small passages, because a full document is too large to feed into the model as context and too coarse to search accurately. Chunking is more subtle than it sounds. Chunks that are too small lose meaning, chunks that are too large blur the retrieval signal, and chunks that ignore natural boundaries like paragraphs or headings tend to cut ideas in half. Overlap between neighbouring chunks helps preserve context that would otherwise be lost at the split.

Second, each chunk is passed through an embedding model, which converts the text into a dense vector that captures its meaning. Similar passages produce similar vectors, and unrelated passages produce vectors that point in different directions. Those vectors are stored in a vector database — FAISS, Chroma, Qdrant, or a hosted equivalent — along with a pointer back to the original chunk.

Third, when a user asks a question, the same embedding model turns the question into a vector, and the vector store returns the chunks whose vectors are closest to it. This is called semantic search, because it retrieves by meaning rather than by exact keyword match. The retrieved chunks are then packed into the prompt in a specific order and passed, together with the question and a system message, to the LLM. The model reads the passages and writes an answer that cites them.

The final step, which is easy to skip and expensive to skip, is handling the case where retrieval fails. If the vector store does not contain anything relevant, a well-designed pipeline refuses to answer instead of letting the model guess. This is the single most effective guard against RAG-flavoured hallucination.

## Retrieval is more than semantic search

Dense embedding search is not the only way to find relevant chunks, and in most serious systems it is not enough by itself. Embeddings are strong on paraphrase and conceptual similarity but weak on rare tokens — exact product names, error codes, acronyms, SKUs — because those tokens carry very little semantic signal in a shared embedding space. The classic remedy is hybrid search, which combines dense retrieval with a sparse keyword retriever such as BM25 and merges the two ranked lists using reciprocal rank fusion or a weighted score. The keyword side of the hybrid catches the exact matches that embeddings miss, and the dense side catches the paraphrases that keyword search misses. A seventy-thirty dense-sparse split is a reasonable starting point for most domains, but the right weighting is empirical and should be tuned against a real evaluation set.

## Re-ranking with cross-encoders

Even after hybrid retrieval, the top candidates are only approximately relevant. Bi-encoder models — the kind used for both dense retrieval and for chunk embedding — score each chunk in isolation and then compare vectors, which is fast but coarse. A cross-encoder, in contrast, takes the query and a candidate chunk together and produces a single relevance score by attending across both at once. It is much more accurate and much more expensive, so it is not used for the initial retrieval; instead, a common pattern is to retrieve twenty or fifty candidates cheaply and then re-rank them with a cross-encoder such as BGE reranker or a hosted Cohere Rerank model, keeping only the top four or five for the final prompt. This two-stage approach captures most of the quality benefits of a giant retriever without the cost.

## Query transformation

The user's question is not always the best query. Multi-query retrieval asks a small LLM to rewrite the original question into two or three alternate phrasings and retrieves for all of them, taking the union of results. HyDE, short for hypothetical document embeddings, has the LLM first draft a plausible answer to the question, then embeds that hypothetical answer and searches with it — often catching passages that share vocabulary with an answer but not with the question. Query decomposition splits a compound question into sub-questions retrieved independently, which helps when a single query would return chunks that satisfy only one part of what the user asked. All three tactics add a small LLM call to the pipeline in exchange for meaningful gains in recall.

## Chunking is a design decision

The simplest chunker splits documents into fixed-size character or token windows with a bit of overlap, and for short, well-formatted markdown this is genuinely good enough. On longer or messier documents it becomes the biggest source of retrieval failure. Semantic chunking uses an embedding model to detect the natural breakpoints in a document and splits there, keeping related sentences together. Header-aware chunking respects the document's own structure, using headings as split boundaries so that each chunk represents one coherent subsection. Proposition-based chunking, a newer idea, extracts atomic factual statements from the source and treats each one as its own chunk; recall goes up dramatically on fact-lookup tasks but the pipeline becomes more complex. Whichever style you pick, giving each chunk a small metadata payload — the source document, the section title, an updated-at timestamp, a tenant or permission tag — pays for itself many times over.

## Long context and RAG

With frontier models now offering context windows of a hundred thousand tokens or more, it is fair to ask whether RAG is being made obsolete. The honest answer is no, at least not yet. Longer contexts still cost more per request, are slower to serve, and are subject to the well-documented "lost in the middle" pattern where information buried inside a huge prompt is used less reliably than information near the top or bottom. Retrieval also gives you access controls, provenance for citations, and the ability to update knowledge without touching a single weight. What long contexts do change is the balance: chunk sizes can be bigger, retrieval can be more permissive, and you can afford to include some surrounding context around each hit. RAG becomes less a compression strategy and more a relevance strategy.

## Evaluating a RAG system

RAG systems fail in two distinct places, and both should be measured separately. Retrieval quality is usually judged with recall at k on a small hand-curated set of query-and-expected-source pairs — did the correct chunk make it into the top four? Answer quality is judged along two axes: faithfulness, which asks whether the generated answer is actually supported by the retrieved chunks and not by the model's imagination, and answer relevance, which asks whether the answer actually addresses the user's question. Tools like Ragas or a homemade LLM-as-judge harness are enough to get started. Skipping this evaluation step turns every subsequent tuning decision — chunk size, hybrid weight, re-ranker choice — into vibes-based guesswork.

## What fine-tuning does, and where it belongs

Fine-tuning is a different intervention entirely. Where RAG changes what the model sees at inference time, fine-tuning changes the model itself. There are several flavours. Full fine-tuning updates every weight in the network and produces the most thorough change, but it requires enormous compute and easily damages the model's general abilities. Supervised fine-tuning trains the model on curated input-output pairs, teaching it to imitate a specific style, format, or reasoning pattern. Instruction tuning is a form of supervised fine-tuning aimed at making a base model good at following instructions. Parameter-efficient methods like LoRA and other adapters add small trainable matrices on top of a frozen model, giving most of the benefit of fine-tuning at a fraction of the cost and without overwriting the original weights.

Fine-tuning is powerful, but it is a poor fit for facts that change over time. If a fine-tuned model has memorised last quarter's pricing and the price changes, the only way to update it is to gather new training data and run the training job again. That is slow, expensive, and it does not scale to knowledge bases that change every day.

## Fine-tuning data is the whole game

The most common mistake in fine-tuning is thinking of it as a scale problem when it is actually a quality problem. A few hundred carefully-written examples of the exact behaviour you want will almost always beat tens of thousands of noisy, contradictory ones scraped together in a rush. Good fine-tuning datasets are small, opinionated, and consistent: every example demonstrates the tone, structure, and reasoning pattern you want the model to imitate, and there are no contradictory examples pulling the model in different directions. It is also useful to include a handful of adversarial or edge-case examples — questions the model should refuse, formats it should not break, boundaries it should hold — because the model will absorb those norms just as reliably as the positive examples. If the target task is genuinely narrow, a LoRA adapter on a strong base model, trained on a few hundred crisp examples for a few epochs, will often beat a full fine-tune trained on much more data.

## Alignment: RLHF, DPO, and RLAIF

Behaviour tuning at the frontier does not stop at supervised fine-tuning. Reinforcement learning from human feedback, usually shortened to RLHF, is the technique that turned raw pretrained models into the assistants people actually enjoy talking to. In RLHF, human annotators rank pairs of model responses, a reward model is trained on those rankings, and the LLM is then optimised against the reward model with a policy-gradient method such as PPO. Direct preference optimisation, or DPO, is a simpler and increasingly popular alternative that skips the explicit reward model and fits the preferences directly with a supervised-style loss. Reinforcement learning from AI feedback, RLAIF, replaces the human annotator with another LLM guided by a written constitution, which is dramatically cheaper and often surprisingly competitive. All three are behaviour interventions, not knowledge interventions — they teach the model how to respond, not what facts to know. The same "fine-tune for behaviour, retrieve for knowledge" rule applies.

## When to prefer RAG, when to prefer fine-tuning

The choice is not aesthetic; it is driven by what you are trying to change about the model's behaviour. RAG is the right tool when the answer depends on information that is frequently updated, tenant-specific, or must be cited. It shines for customer-support bots that need to quote a policy document, internal search assistants that answer questions about a company wiki, and legal or medical assistants that need every claim traceable to a source. RAG also allows access controls: because retrieval is a separate step, you can restrict which chunks a user is allowed to see before the model ever gets involved.

Fine-tuning is the right tool when the answer depends on how the model behaves rather than what it knows. If a model needs to adopt a specific tone, follow a strict output format, understand a specialised vocabulary that never appears in general web text, or perform a narrow task with lower latency than a long RAG prompt would allow, fine-tuning is often the cleaner solution. It bakes the desired behaviour into the weights so no retrieval step is needed at inference time.

## Combining the two

In practice, mature production systems use both. A model is fine-tuned to speak in the right voice, follow the right output schema, and handle domain-specific vocabulary, and then RAG is layered on top to supply the up-to-date facts. The two techniques are complementary rather than competitive. The rule of thumb is simple: fine-tune for behaviour, retrieve for knowledge.
