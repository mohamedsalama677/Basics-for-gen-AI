# Agentic AI

## What people mean by "agentic"

Agentic AI is the term that describes systems in which a language model is not used as a one-shot text generator but as the decision-maker inside a longer loop. In a plain LLM call, the user asks a question, the model answers, and the interaction ends. In an agentic system, the model is given a goal, a set of tools it can use, and the freedom to decide, step by step, which tool to call and when to stop. The model's outputs are no longer just words for a human to read; some of them are actions that change the state of the world, and some of them are observations of what happened after those actions were taken. This shift, from generation to iterated decision-making, is what makes something feel like an agent rather than a chatbot.

## The core loop

Almost every agentic system, no matter how it is dressed up, follows the same underlying loop. The agent perceives its current situation, which usually means reading the user's request together with any results from previous steps. It then plans what to do next, which in practice means the LLM generates a decision, often expressed as a tool call in a structured format. The chosen action is executed by the surrounding runtime — this is the part that is not the LLM: a Python function runs, an API is called, a database is queried, a file is written. The result of that action is observed, added back into the model's context, and the loop starts over. The agent keeps cycling until it decides that the goal has been reached, or until an external limit stops it.

The mechanics of this loop matter less than the fact that it exists. Once you have a loop, the model can attempt something, see whether it worked, and try a different approach if it did not. That capacity to correct itself is where most of the value of agents comes from, and it is also where most of the risks come from.

## Tools and function calling

Tools are the interface between the language model and the outside world. Modern chat-tuned models can be given a list of tool definitions — each one a name, a natural-language description, and a schema for its arguments — and the model will emit structured tool calls when it thinks a tool is needed. This is usually called function calling. Under the hood, the model is still just generating tokens, but the tokens are constrained to match the tool schema, so the surrounding runtime can parse them reliably and dispatch the actual work. Good tool design matters more than most first-time builders realise. Descriptions should be unambiguous, arguments should be typed, and the return value should be short enough for the model to read without blowing the context window. A tool that returns a two-megabyte JSON blob is worse than useless, because the model will drown in it.

## Planning strategies

Different agentic systems use different planning styles. The simplest, and still one of the most common, is ReAct, short for Reasoning and Acting. In ReAct the model interleaves short chains of reasoning with tool calls, thinking briefly about what to do, taking one action, observing the result, and thinking again. This is a tight loop that reacts to feedback quickly but can wander if the task is long. Plan-and-execute is a different pattern: the model first writes an explicit plan, a numbered list of steps, and then executes them one by one, only re-planning if something goes wrong. This tends to be more efficient on well-defined tasks but is brittle when reality does not match the plan. More sophisticated systems use hierarchical planning, where a top-level agent decomposes a goal into sub-goals and delegates each one to a sub-agent.

## Memory

Because the loop can run for many steps, memory becomes a first-class concern. Short-term memory is the current context window, which holds the recent conversation and the running trace of actions and observations. It is fast, but it is finite, and long agent runs will overflow it. Long-term memory is stored outside the model, usually in a vector database or a structured store, and it is retrieved back into the prompt when relevant. This is essentially RAG applied to the agent's own history rather than to external documents. Well-designed agents also have working memory for intermediate results — scratch space that is not shown to the user but is used to track progress across steps.

## Multi-agent systems

Once one agent works, the obvious next question is whether several agents can cooperate. Multi-agent systems assign different roles to different LLM instances — one plans, one writes code, one reviews, one talks to the user — and let them communicate through a shared message bus or a central coordinator. This can produce impressive results on complex tasks, because specialisation lets each agent have a tighter system prompt and a smaller tool set. It also multiplies the risks: agents can talk in circles, hallucinate at each other, and rack up costs at surprising speed. In practice, most production systems are simpler than the multi-agent literature suggests, and a single well-instrumented agent usually beats a poorly-orchestrated committee.

## Safety and evaluation

Agentic systems are harder to make safe than plain chatbots because they take actions. A model that hallucinates a fact is embarrassing; a model that hallucinates a database write is expensive. Prompt injection is especially dangerous in this setting, because a malicious document retrieved by the agent can contain instructions that the model may follow. Sensible defences include restricting the surface area of each tool, requiring human confirmation for irreversible actions, sandboxing code execution, and enforcing hard limits on cost and step count.

Evaluation is also harder. There is no single correct answer to grade against, because the same goal can be reached by different action sequences. Teams end up combining traditional metrics with trajectory-level evaluation, checking whether the agent reached the goal, how many steps it took, whether it used the right tools, and whether it avoided obviously bad actions along the way.

## When to reach for an agent, and when not to

The honest answer is that most tasks do not need an agent. If the workflow is a fixed pipeline of steps, a plain chain is cheaper, faster, and easier to debug. Agents earn their keep when the number of steps is genuinely unknown ahead of time, when the right next action depends on what the previous action returned, or when the tool space is large enough that pre-scripting every branch is impractical. Reserve them for those cases, and the loop starts to feel less like a party trick and more like a durable pattern.

## Frameworks in the wild

The agent space has coalesced around a handful of frameworks, each making different bets about how much structure to impose. LangGraph, from the LangChain team, models agents as explicit state machines with typed nodes and edges, which makes it easy to reason about control flow and easy to add human-in-the-loop steps or conditional routing. AutoGen, from Microsoft Research, leans into multi-agent conversations, letting several role-specialised agents chat until they reach an answer. CrewAI takes a similar multi-agent angle but with a more opinionated "team of workers" abstraction on top. The OpenAI Assistants API and Anthropic's Claude Agent SDK provide provider-hosted runtimes where the model, memory, and tool dispatch are wrapped together behind a simpler surface. There is no single right choice, and the best framework depends on how much you want to own versus how much you want the framework to hide. For voice agents specifically, the LiveKit Agents SDK is the current standard, because it handles the real-time media pipeline and turn-taking that a general agent framework does not.

## Structured output and JSON mode

Function calling is one way to get structured data out of a model, but sometimes you want a structured response even when no tool is being called. Every major provider now supports a mode that constrains the model's output to a supplied JSON schema — OpenAI calls it structured outputs, Anthropic exposes it through tool use with a strict flag, and open-source models achieve it through libraries like Instructor or Outlines that constrain the sampler at decoding time. The value of this mode is downstream reliability: instead of parsing free-form text with brittle regexes, you receive a validated object that your code can consume directly. In agent runtimes, this is what turns model output into something safe to feed back into the loop without a validation layer of its own.

## Reflection and self-critique

A single-shot agent decision is often wrong in ways the agent itself can spot if given the chance. Reflection patterns exploit this by adding a second LLM call — sometimes the same model, sometimes a stronger critic — that reviews the proposed action or draft answer before it is executed or returned. If the critic finds a problem, the agent tries again with the critique added to context. This is closely related to self-consistency (sampling several attempts and taking the majority answer) and to more elaborate patterns like Reflexion, which lets the agent write persistent lessons-learned notes across attempts. Reflection is powerful on reasoning-heavy tasks, but it is not free — each critique doubles the LLM cost of that step, and there is a real risk of the critic being wrong too. Use it where the failure cost is high and the base agent is known to be brittle.

## Tool selection at scale

A handful of tools is trivial to manage: put them all in the system prompt and let the model choose. When the tool count grows into the dozens or hundreds — plausible in enterprise settings with many internal APIs — this stops working. The model's context fills up with tool schemas, latency climbs, and the model gets worse at picking the right tool as options multiply. The standard mitigation is tool retrieval: embed the tool descriptions, embed the user's request, and only surface the top handful of relevant tools to the model on each turn. Some systems go further and organise tools into a hierarchy, where the model first picks a category and then, in a second turn, picks a specific tool within it. Either way, treating the tool list as a searchable knowledge base rather than a fixed prompt is what makes large tool spaces tractable.

## Cost and latency management

An agentic system with several LLM calls, one or two tool round-trips, and maybe a reflection step can easily cost ten times what a single-shot chat call would cost, and be five times slower. Production agents survive by aggressively managing both. Prompt caching, where a fixed system prompt is charged only once even across many follow-up turns, is now offered by every major provider and is essentially free to enable. Model tiering routes cheap or easy subtasks to a smaller model — for example, using a small classifier to decide whether a call to the expensive planner is even needed. Batching accumulates multiple independent requests and sends them together to amortise per-request overhead. Result caching, keyed on prompt-plus-parameters hash or on the semantic similarity of prompts, catches the huge fraction of production traffic that is essentially duplicate work. None of these are exotic; skipping them is what makes many agent proofs-of-concept economically unshippable.

## Human-in-the-loop patterns

Fully autonomous agents are appropriate for a narrow slice of use cases. For anything with real consequences — writing to production databases, sending customer emails, executing financial transactions — the sane pattern is a human in the loop. This can be as light-touch as an "approve or reject" step before a specific class of action, or as heavy as a full trajectory review where a human inspects the agent's proposed plan before execution begins. LangGraph and similar state-machine frameworks make this pattern explicit by allowing execution to pause on a specific node and wait for external input. The tricky part is designing the surface: humans get bored of approving obviously-fine actions and start rubber-stamping, which destroys the value of the review. Reserving human approval for the actions where a mistake would be irreversible, and letting the safe ones run automatically, is the usual compromise.

## Observability and tracing

Debugging an agent by reading logs is painful because the LLM's decisions are opaque and the failure often happened three steps ago. Modern agent stacks are almost always paired with a tracing tool such as LangSmith, LangFuse, Arize, or a plain OpenTelemetry setup, which records every LLM call, tool call, retry, and intermediate state along with the exact prompts and responses. When something goes wrong in production, the fastest path to a fix is usually to open the trace, find the step where the agent went off the rails, and see whether the model got bad input, misinterpreted a tool result, or hallucinated an action. Building this observability in from the first prototype pays back many times over as soon as anything reaches real users, because agent bugs are almost always narrative bugs — they only make sense when you can watch the whole story unfold.

## Voice agents specifically

Voice agents are a specialisation of the general agent pattern with a real-time media pipeline underneath. The standard architecture pipes microphone audio through a voice-activity detector to segment speech from silence, then through a streaming speech-to-text model to get a live transcript, then into an LLM that may call tools mid-turn, and finally through a text-to-speech model whose output is streamed back to the speaker. Everything has to run at conversational latency, which usually means end-to-end response times under two seconds, and every stage has to support interruption. Barge-in support — the ability for a user to talk over the agent and be listened to — is what separates a natural-feeling voice agent from a walkie-talkie. The system must be able to detect that the user started speaking mid-response, cancel the ongoing text-to-speech stream, and update the model's context so the next turn is aware only of what the user actually heard. RAG and tool calls fit into this loop exactly as they do in a text agent, but every extra LLM round trip is felt directly in the perceived latency, so voice agents put unusual pressure on cache-friendliness and cheap intermediate models.

## Browser and computer-use agents

The newest and most ambitious agents do not just call APIs — they operate general-purpose software. Browser-use agents drive a real browser by observing rendered pages and clicking, typing, and scrolling. Computer-use agents extend this to the whole desktop, reading pixel screenshots and issuing mouse and keyboard actions. Anthropic and OpenAI have both released production versions of this pattern, and open-source projects have followed. The appeal is that these agents can operate any software, including systems that never exposed an API. The catch is that they are slow, expensive, and easily distracted by unexpected UI states, and they concentrate the safety risks of agentic AI in one place: a single hallucinated action can now be a click on a payment button. This is a promising but immature space, and the current best practice is to keep computer-use agents heavily sandboxed and human-supervised until the failure modes are better understood.
